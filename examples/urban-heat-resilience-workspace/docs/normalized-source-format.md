# Normalized Source Format

Normalized source records are agent-readable Markdown files generated from manifest records. They live under `sources/normalized/` and provide the stable input layer for source notes, claim extraction, local search, and linting.

Raw files stay immutable under `raw/`. Normalized files may be regenerated by normalization tooling. Curated interpretation belongs in `wiki/sources/`, not in normalized records.

## File Path

Use one normalized file per manifest source ID:

```text
sources/normalized/<safe-source-id>.md
```

`<safe-source-id>` is the manifest `id` made filesystem-safe:

- replace `:` with `--`;
- replace `/` and whitespace with `-`;
- keep lowercase letters, numbers, `.`, `_`, and `-`;
- collapse repeated `-`.

Examples:

| Source ID | Normalized Path |
|-----------|-----------------|
| `paper:2604.13018v1` | `sources/normalized/paper--2604.13018v1.md` |
| `link:github-aweai-team-aiscientist-a1b2c3d4e5` | `sources/normalized/link--github-aweai-team-aiscientist-a1b2c3d4e5.md` |
| `manual:lab-notes-agent-evals` | `sources/normalized/manual--lab-notes-agent-evals.md` |

## Frontmatter

Every normalized record must start with YAML frontmatter.

Required fields:

```yaml
---
type: normalized_source
source_id: paper:2604.13018v1
source_kind: paper
status: content_extracted
created: 2026-05-09
updated: 2026-05-09
raw_paths:
  - raw/other/arXiv-2604.13018v1
  - raw/pdf/2604.13018v1.pdf
manifest_path: sources/manifest.jsonl
normalizer:
  name: normalize_sources.py
  version: 1
parse_warnings: []
---
```

Required field meanings:

| Field | Meaning |
|-------|---------|
| `type` | Always `normalized_source`. |
| `source_id` | Exact manifest `id`, or a stable manual ID for manual records. |
| `source_kind` | Manifest `kind`, such as `paper`, `pdf`, `repo_link`, `web_link`, `codebase_architecture`, or `manual_note`. |
| `status` | Normalization lifecycle status. |
| `created` | Date the normalized record was first created, `YYYY-MM-DD`. |
| `updated` | Date the normalized record was last updated, `YYYY-MM-DD`. |
| `raw_paths` | Workspace-relative raw paths used as evidence. Empty only for manual records. |
| `manifest_path` | Workspace-relative manifest path, usually `sources/manifest.jsonl`. |
| `normalizer` | Tool name and version that produced the record. |
| `parse_warnings` | List of non-fatal extraction issues. Use `[]` when clean. |

Allowed `status` values:

| Status | Meaning |
|--------|---------|
| `stubbed` | Source is represented but content was not fetched or extracted. |
| `content_extracted` | Main source content was extracted into Markdown. |
| `partial` | Some content was extracted, but important parts are missing or unreliable. |
| `failed` | Extraction failed; the record exists to preserve evidence and warnings. |

Optional frontmatter fields:

```yaml
title: Toward Autonomous Long-Horizon Engineering for ML Research
authors:
  - Example Author
venue: arXiv
date: 2026-04-01
doi:
arxiv_id: 2604.13018v1
url: https://arxiv.org/abs/2604.13018
raw_pdf: raw/pdf/2604.13018v1.pdf
latex_root: raw/other/arXiv-2604.13018v1
entrypoint: main.tex
repo_full_name: AweAI-Team/AiScientist
codebase_repo:
codebase_revision:
codebase_tool:
codebase_artifact_paths: []
extraction_method: latex
content_hash: sha256:...
language: en
confidence: high
```

Optional field meanings:

| Field | Meaning |
|-------|---------|
| `title`, `authors`, `venue`, `date`, `doi`, `arxiv_id`, `url` | Citation metadata when available. |
| `raw_pdf`, `latex_root`, `entrypoint` | Source-specific raw locations from the manifest. |
| `repo_full_name` | GitHub owner/repository for repository links. |
| `codebase_repo`, `codebase_revision`, `codebase_tool`, `codebase_artifact_paths` | Optional metadata for normalized codebase architecture records. |
| `provider` | Inferred provider for link stubs, such as `github` or a URL host. |
| `fetch_status` | Fetch or artifact state for link and codebase stubs. Use `not_fetched` when no network content has been fetched and `artifact_recorded` when a local codebase artifact was normalized. |
| `extraction_method` | Method used, such as `latex`, `pdf_text`, `link_stub`, `manual`, `web_stub`, `codebase_context`, or `codebase_stub`. |
| `content_hash` | Hash of normalized extracted content for reproducibility. |
| `language` | BCP-47 language tag when known. |
| `confidence` | `high`, `medium`, or `low` extraction confidence. |

## PDF Text Fallback Dependency

PDF-only records may be normalized with `extraction_method: pdf_text` when no usable LaTeX entrypoint exists. The fallback uses the local Poppler `pdftotext` command:

```text
pdftotext -layout -enc UTF-8 <pdf> -
```

If `pdftotext` is not installed, normalization must fail with an action message instead of silently adding or downloading a dependency. Install Poppler through the operating system package manager, such as `poppler-utils` on Debian/Ubuntu.

## Codebase Architecture Records

Codebase analysis is optional. Use `source_kind: codebase_architecture` when a repository, source archive, or implementation snapshot is research evidence.

Recommended rules:

- use read-only analysis commands when possible;
- store generated architecture output under `sources/`, not directly under the maintained `wiki/`;
- cite codebase analysis through source notes and `source_ids`;
- include `codebase_repo`, `codebase_revision`, and `codebase_tool` when known;
- preserve tool warnings and unsupported-language gaps in `parse_warnings`;
- do not install hooks or background sync while normalizing codebase evidence.

For `agent-wiki-cli` / `python-wiki-llm`, prefer `llm-wiki extract` and `llm-wiki context` output for normalized records. Use generated architecture wiki output only as a source artifact, for example under `sources/code_wikis/<source_id>/`.

`normalize_sources.py` reads local adapter artifacts only. It looks for preferred files such as `context.json`, `extract.json`, `summary.json`, `context.md`, `extract-summary.md`, or `summary.md` in the manifest record's `metadata.codebase_output_dir`. If no artifact exists, the normalizer writes a `codebase_stub` with a warning and the configured command as a manual next action. It never runs `llm-wiki`, clones repositories, fetches network content, or syncs a generated architecture wiki.

## Normalization CLI

Run normalization from the research workspace root:

```bash
python3 scripts/normalize_sources.py
```

Default behavior normalizes only pending eligible manifest records, meaning records whose normalized Markdown file does not already exist.

Common controls:

| Command | Behavior |
|---------|----------|
| `python3 scripts/normalize_sources.py` | Create normalized records for pending eligible sources only. |
| `python3 scripts/normalize_sources.py --source-id paper:2604.13018v1` | Normalize one source ID. |
| `python3 scripts/normalize_sources.py --source-id ID1 --source-id ID2` | Normalize a selected subset. |
| `python3 scripts/normalize_sources.py --all` | Consider all eligible sources, skipping existing outputs. |
| `python3 scripts/normalize_sources.py --all --force` | Regenerate every eligible source. |
| `python3 scripts/normalize_sources.py --all --dry-run` | Print planned actions without extraction or writes. |
| `python3 scripts/normalize_sources.py --append-log` | Append a compact run summary to `log.md`. |

Existing normalized records are never overwritten unless `--force` is supplied.
Use `--force` with `--source-id` for a precise refresh or with `--all` for a complete regeneration. `--append-log` is skipped during `--dry-run`.

Each run prints a stderr summary with selected, planned, created, updated, skipped, dry-run, method, partial, and failed counts.

## Required Sections

Use these sections in this order. If a section has no content, write `None extracted.` or `None recorded.` rather than deleting the section.

```markdown
# <title or source_id>

## Citation Metadata

## Abstract

## Outline

## Extracted Text

## Figures and Tables

## Links

## Raw Source Paths

## Parse Warnings
```

Section expectations:

| Section | Content |
|---------|---------|
| `Citation Metadata` | Human-readable citation fields copied from frontmatter or extracted text. |
| `Abstract` | Source abstract, summary, or `None extracted.` |
| `Outline` | Section hierarchy or source structure. |
| `Extracted Text` | Main normalized source text. Preserve headings when possible. |
| `Figures and Tables` | Figure/table references, captions, labels, and extraction notes. |
| `Links` | URLs, repository links, arXiv links, project pages, or related raw links. |
| `Raw Source Paths` | Evidence paths from `raw_paths` plus any manifest source-specific paths. |
| `Parse Warnings` | Same warnings as frontmatter, with enough context for review. |

## Citation From Wiki Source Notes

Wiki source notes cite normalized records through `source_ids`:

```yaml
---
type: source
created: 2026-05-09
updated: 2026-05-09
source_ids:
  - paper:2604.13018v1
---
```

When a source note references `paper:2604.13018v1`, tooling should resolve the normalized record at:

```text
sources/normalized/paper--2604.13018v1.md
```

Source notes may link to normalized records in body text, but `source_ids` is the canonical machine-readable citation bridge.

## Validation Expectations

Future linting should check:

- required frontmatter fields exist;
- `type` is `normalized_source`;
- `source_id` matches a manifest record unless it starts with `manual:`;
- required sections are present in order;
- `parse_warnings` frontmatter and the `Parse Warnings` section are consistent;
- wiki source notes cite normalized records through `source_ids`.
